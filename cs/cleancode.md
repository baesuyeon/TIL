# 클린 코드 정리

장인 정신을 익히는 두 단계
1. 이론

원칙, 패턴, 기법, 경험이라는 지식을 습득해야한다.
좋은 코드에 대한 기준과 방법론을 꾸준히 학습해야 한다.

2. 실전

열심히 일하고 연습해 지식을 체득해야 한다.
고생을 해야한다 (실패도 맛봐야한다)

## 깨끗한 코드
**르블랑의 법칙**
> 나중은 결코 오지 않는다.
일단 돌아가게만 만들고 나중에 정리하자는 생각은 대부분 실현되지 않는다.

**나쁜 코드는 개발 속도를 크게 떨어뜨린다.**
기한을 맞추는 유일한 방법, 빨리 가는 유일한 방법은 언제나 코드를 최대한 깨끗하게 유지하는 습관이다.

**깨끗한 코드란**
* 의도가 명확하다.
* 읽기 쉽다.
* 한가지 역할에 집중한다.
* 중복이 적다.
* 세부 사항까지 꼼꼼하다.

예를 들어 다음 요소들도 모두 코드 품질에 포함된다.
* 오류 처리
* 메모리 누수
* 경쟁 상태(Concurrency)
* 일관성 있는 네이밍
* 예외 상황 처리

깨끗한 코드는 세세한 사항까지 꼼꼼하게 정리하는 코드다 (오류처리, 메모리 누수, 경쟁상태, 일관성 없는 명명법)
깨끗한 코드는 단순하고 한가지에 집중한다.

**읽기 쉬운 코드가 중요한 이유**
우리는 새로운 코드를 작성하는 시간보다 기존 코드를 읽는 시간이 훨씬 많다.
새 코드를 짜면서 우리는 끊임없이 기존 코드를 읽기 때문에 읽기 쉬운 코드가 매우 중요하다.

**지속적인 개선**
한번에 많은 시간과 노력을 투자해 코드를 정리할 필요가 없다.
변수 이름 하나를 개선하고, 조금 긴 함수 하나를 분할하고, 약간의 중복을 제거하고, 복잡한 if 문 하나를 정리하면 충분하다.

**좋은 개발자란**
빈 캔퍼스를 우아한 작품으로 바꿔가는 화가와 같다.

# 의미있는 이름
**의도가 드러나는 이름을 사용하라**

이름만 보고도 다음을 알 수 있어야 한다.
* 왜 존재하는가?
* 무엇을 하는가?
* 어떻게 사용되는가?

주석이 필요하다면 이름이 충분히 설명적이지 못할 가능성이 크다.

나쁜 예시
```java
int d; // 경과 시간(단위: 날짜) ❌
```
변수명만 봐서는 의미를 전혀 알 수 없다.

좋은 에시
```java
int elapsedTimeInDays;     // ⭕ 경과 시간(일 단위)
int daysSinceCreation;     // ⭕ 생성 이후 지난 일수
int daysSinceModification; // ⭕ 수정 이후 지난 일수
int fileAgeInDays;         // ⭕ 파일의 경과 일수
```
이름만 봐도 변수의 역할과 의미가 명확하다.

**의미 있게 구분하라**
이름은 서로 명확히 구분되어야 한다.

나쁜 예시
```text
a1, a2
data, info
object
```

좋은 예시
```text
source
destination
```

## 인코딩을 피하라
타입 정보를 이름에 넣지 말자.

나쁜 예시
```text
String strName
IShapeFactory
```
특히 인터페이스에 I 접두어를 붙이는 방식은 가독성을 떨어뜨릴 수 있다.

# 함수
프로그램의 가장 기본적인 단위가 함수다.

## 작게 만들어라
작은 함수는
* 읽기 쉽고
* 테스트하기 쉽고
* 재사용하기 쉽다.

## 더 작게 만들어라

## 블록과 들여쓰기
if / else / while 내부 블록은 가능한 단순해야 한다.
이상적인 블록은 함수 호출 한 줄 정도다.

## 한가지만 해라
함수는 반드시 하나의 역할만 수행해야 한다.

**중요한 기준**
함수 내부에서 의미 있는 이름으로 또 다른 함수를 추출할 수 있다면
현재 함수는 여러 일을 하고 있을 가능성이 크다.

## 함수당 추상화 수준은 하나로 유지하라
`getHtml()` → 매우 높은 수준의 추상화
`String pagePathName = PathParser.render(pagepath);` → 중간 수준
`.append("\n")` → 매우 낮은 수준

한 함수 내에 추상화 수준을 섞으면 코드를 읽는 사람이 헷갈린다.

## 서술적인 이름
길고 서술적인 이름이 짧고 어려운 이름보다 좋다.
* calculateMonthlySubscriptionPrice()

## 함수 인수
인수는 적을수록 좋다
이상적인 함수는 인수가 없다.

* 1개: 괜찮음
* 2개: 가능
* 3개: 신중
* 4개 이상: 특별한 이유 필요

인수가 많아질수록 이해와 테스트가 어려워진다.
웬만하면 단항 함수로 바꾸도록 애써야 한다.

## 플래그 인수
플래그 인수는 추하다.
true가 무엇을 의미하는지 알기 어렵다.
플래그 인수는 함수가 여러 역할을 수행한다는 신호이기도 하다.

나쁜 예시
```java
render(true);
```

## 인수 객체
인수가 많다면 관련 데이터를 객체로 묶어 인수를 줄일 수 있다.

## 동사와 키워드
함수 이름에 인수 이름을 넣으면 가독성이 좋아진다.

```java
writeField(name);
```

## 부수효과를 일으키지 말라
함수는 이름에 드러난 일만 해야 한다.

위험한 예시
* 전역 변수 수정
* 전달받은 객체 변경
* 세션 상태 변경

이런 동작은 예측하기 어려운 버그를 만든다.
함수가 한가지만 한다는 규칙을 위반하지만 `checkPasswordAndInitializeSession` 이라는 이름이 훨씬 좋다.

## 출력 인수를 사용하지 말라
출력 인수는 함수의 반환값(return) 대신 전달받은 인자를 직접 수정해서 결과를 만들어내는 방식을 말한다.

나쁜 예시
```java
appendFooter(report);
```
이 함수가 report에 footer를 추가하는지, footer에 report를 추가하는지 명확하지 않다.

```java
public void appendFooter(StringBuffer report);
```
선언부를 직접 확인해야 한다.

좋은 예시
```java
report.appendFooter();
```
객체 스스로 자신의 상태를 변경하도록 만드는 편이 훨씬 자연스럽고 읽기 쉽다.

## 명령(Command)과 조회(Query)를 분리하라
함수는 아래 둘 중 하나만 해야 한다.
* 명령(Command): 객체의 상태를 병ㄴ경한다.
* 조회(Query): 정보를 반환한다.

두 역할을 동시에 수행하면 함수의 동작을 예측하기 어려워진다.

## 오류 코드보다 예외(Exception)를 사용하라
과거에는 함수가 실패했을 때
오류 코드를 반환하는 방식이 자주 사용되었다.

이 방식의 문제점은 호출하는 쪽이 매번 오류 처리를 직접 해야 한다는 점이다.
결국 비즈니스 로직보다 오류 처리 코드가 더 많아질 수 있다.

예외를 사용하면 흐름이 단순해진다.

## try/catch 블록은 별도 함수로 분리하는 것이 좋다
try/catch 블록은 코드 가독성을 크게 떨어뜨릴 수 있다.
따라서 가능하면 별도 함수로 분리하는 것이 좋다.

```java
public void delete(Page page) {
    try {
        deletePageAndAllReferences(page);
    } catch (Exception e) {
        logError(e);
    }
}
```
호출부에서는 삭제 작업을 수행하는구나 정도만 읽고 넘어갈 수 있다.

오류 처리도 한가지 작업이다.
함수에 키워드 try가 있다면 함수는 try로 시작해 catch/finally 문으로 끝나야 한다.

## 오류 코드(Enum) 방식의 또 다른 문제
오류코드를 반환한다는 이야기는 어디선가 오류 코드를 정의한다는 뜻이다.
다른 클래스들은 모두 ErrorCode enum에 의존하게 된다.
새로운 오류를 추가할 때 switch 문 처리가 누락될 가능성도 있고 클라이언트 코드 수정이 필요할 수도 있다. → 결합도가 높아진다.

예외 방식은 좀 더 유연하다
```java
class InvalidPasswordException extends Exception
```
새로운 예외를 추가해도 기존 코드를 크게 건드리지 않아도되고
새로운 오류 상황을 확장하기 쉬워진다.

# 주석
> 좋은 코드는 주석으로 설명하는 코드가 아니라 주석 없이도 읽히는 코드다

주석은 오래될수록 코드에서 멀어진다.
코드는 변화하고 진화한다.
부정확한 주석은 아예 없는 주석보다 훨씬 나쁘다.

## 코드로 의도를 표현하라
좋은 코드는 주석 없이도 읽혀야 한다.

나쁜 예시
```java
// 직원에게 복지 혜택을 받을 자격이 있는지 검사한다.
if((employee.flags & HOURLY_FLAG) && (employee.age > 65))
```

좋은 예시
```java
if (employee.isEligibleForFullBenefits())
```
주석으로 달려는 설명을 함수로 만들어 표현해도 충분하다.
주석을 달고싶을 때는 함수 추출, 변수 이름 개선, 조건 분리 등의 리팩토링을 먼저 고민해보자

## 좋은 주석
좋은 주석은 왜를 설명한다.

```java
// 외부 결제 시스템 버그로 인해 0원 결제 요청은 차단한다.
```
이런 맥락 정보는 코드만으로 표현하기 어렵다.

## 나쁜 주석
* 코드 내용을 그대로 설명하는 주석
* 불필요한 Javadocs
 * 공개 API가 아닌 시스템 내부에 속한 클래스와 함수에 Javadocs를 생성할 필요는 없다.
* 닫는 괄호 주석 `} //`
  * 주석을 달기보다 함수를 분리하여 블록을 줄이고 구조를 개선하라

주석을 추가하기 전에 코드 자체를 더 명확하게 만들 수 없는지 먼저 고민하자.

# 형식 맞추기
* 변수는 처음 사용하는 위치에 최대한 가까이 선언한다.
* 인스턴스 변수는 클래스 맨 처음에 모아 선언한다.
* 한 함수가 다른 함수를 호출한다면 두 함수는 세로로 가까이 배치한다.
  * 호출하는 함수가 먼저오도록 배치하라

# 객체와 자료구조
객체(Object)는
* 데이터를 숨기고 (캡슐화)
* 행동을 공개한다.

자료구조(DTO)는
* 데이터를 공개하고
* 별다른 행동이 없다.

여기서 말하는 추상화는 데이터가 어떻게 저장되는지를 노출하는 것이 아닌 **무엇을 하는지**를 노출하는 것이다.

좋지 않은 예시
```java
class User {
    private String name;

    public String getName();
    public void setName(String name);
}
```
겉보기에는 캡슐화처럼 보이지만 실제로는 내부 데이터를 getter/setter로 그대로 외부에 노출하고 있다.
객체도 자료구조도 아닌 애매한 상태 (비공개 변수를 외부에 노출)

구현 정보를 노출하는 인터페이스
```java
public interface Vehicle {
    double getFuelTankCapacityInGallons();
    double getGallonsOfGasoline();
}
```
getter를 제공하는데 내부 데이터를 직접 다루고 있다. (추상화 수준이 낮고 구현 정보 노출이 강하다)
외부가 내부 데이터를 직접 조합해서 계산하도록 만든다.

좋은 추상화 예시
```java
public interface Vehicle {
    double getPercentFuelRemaining();
}
```
객체 스스로가 의미있는 정보를 제공한다.

## 디미터 법칙
디미터 법칙: 데이터를 꺼내서 처리하지 말고 객체에게 직접 일을 시켜라

객체는 낯선 객체를 많이 알면 안 된다. 친구랑만 놀아라
> 객체 내부를 연쇄적으로 탐색하지 말라.

```java
String outputDir = ctxt.getOptions().getScratchDir().getAbsolutePath();
```
위 코드에서는 options, scratchDir, absolutePath 구조까지 모두 알고있다. (기차 충돌)
이런 구조는 객체들이 강하게 결합되어 내부 구조 변경에 취약하다.

객체에게 직접 일을 시켜라 (만약 자료구조라면 상관이 없다)

ctxt에게 뭔가를 하라고 시킬 수 있다.

```java
BufferedOutputStream bos = ctxt.createScratchFileStream(classFileName);
```
객체에게 임시파일을 생성하라고 시킨다.
호출부에서는 자신이 몰라야하는 여러 객체를 탐색할 필요가 없다.
