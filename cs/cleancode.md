1. 이론

원칙, 패턴, 기법, 경험이라는 지식을 습득해야한다.
좋은 코드에 대한 기준과 방법론을 꾸준히 학습해야 한다.

2. 실전

열심히 일하고 연습해 지식을 체득해야 한다.

## 깨끗한 코드
**르블랑의 법칙**
> 나중은 결코 오지 않는다.
일단 돌아가게만 만들고 나중에 정리하자는 생각은 대부분 실현되지 않는다.

**나쁜 코드는 개발 속도를 크게 떨어뜨린다.**
기한을 맞추는 유일한 방법, 빨리 가는 유일한 방법은 언제나 코드를 최대한 깨끗하게 유지하는 습관이다.

**깨끗한 코드란**
* 의도가 명확하다.
* 읽기 쉽다.
* 한가지 역할에 집중한다.
* 중복이 적다.
* 세부 사항까지 꼼꼼하다.

예를 들어 다음 요소들도 모두 코드 품질에 포함된다.
* 오류 처리
* 메모리 누수
* 경쟁 상태(Concurrency)
* 일관성 있는 네이밍
* 예외 상황 처리

깨끗한 코드는 세세한 사항까지 꼼꼼하게 정리하는 코드다 (오류처리, 메모리 누수, 경쟁상태, 일관성 없는 명명법)
깨끗한 코드는 단순하고 한가지에 집중한다.

**읽기 쉬운 코드가 중요한 이유**
우리는 새로운 코드를 작성하는 시간보다 기존 코드를 읽는 시간이 훨씬 많다.
새 코드를 짜면서 우리는 끊임없이 기존 코드를 읽기 때문에 읽기 쉬운 코드가 매우 중요하다.

**지속적인 개선**
한번에 많은 시간과 노력을 투자해 코드를 정리할 필요가 없다.
변수 이름 하나를 개선하고, 조금 긴 함수 하나를 분할하고, 약간의 중복을 제거하고, 복잡한 if 문 하나를 정리하면 충분하다.

**좋은 개발자란**
빈 캔퍼스를 우아한 작품으로 바꿔가는 화가와 같다.

# 의미있는 이름
**의도가 드러나는 이름을 사용하라**

이름만 보고도 다음을 알 수 있어야 한다.
* 왜 존재하는가?
* 무엇을 하는가?
* 어떻게 사용되는가?

주석이 필요하다면 이름이 충분히 설명적이지 못할 가능성이 크다.

나쁜 예시
```java
int d; // 경과 시간(단위: 날짜) ❌
```
변수명만 봐서는 의미를 전혀 알 수 없다.

좋은 에시
```java
int elapsedTimeInDays;     // ⭕ 경과 시간(일 단위)
int daysSinceCreation;     // ⭕ 생성 이후 지난 일수
int daysSinceModification; // ⭕ 수정 이후 지난 일수
int fileAgeInDays;         // ⭕ 파일의 경과 일수
```
이름만 봐도 변수의 역할과 의미가 명확하다.

**의미 있게 구분하라**
이름은 서로 명확히 구분되어야 한다.

나쁜 예시
```text
a1, a2
data, info
object
```

좋은 예시
```text
source
destination
```

## 인코딩을 피하라
타입 정보를 이름에 넣지 말자.

나쁜 예시
```text
String strName
IShapeFactory
```
특히 인터페이스에 I 접두어를 붙이는 방식은 가독성을 떨어뜨릴 수 있다.

## 클래스 이름
클래스 이름은 명사 또는 명사구를 사용한다.
* User
* PaymentService
* OrderRepository

## 메서드 이름
메서드 이름은 동사 또는 동사구를 사용한다.
* save()
* findUser()
* calculatePrice()

접근자 메서드는 일반적으로 다음 접두어를 사용한다.
* get
* set
* is

## 한 개념에 한 단어를 사용하라
동일한 동작에 여러 단어를 섞어 쓰면 혼란을 만든다.
* fetch
* retrieve
* get
프로젝트 전체에서 일관된 용어를 사용한다. (모두 get, 모두 find)

## 기술 용어를 두려워하지 말라
코드를 읽을 사람도 개발자다.
따라서 적절한 기술 용어는 오히려 이해를 돕는다.
* Queue
* Factory
* Repository

## 의미있는 맥락을 추가하라
firstName, lastName, street, houseNumber, city, state, zipcode 변수 전체를 보면
state가 주소의 일부라는 사실을 알지만 state만 보면 주소의 일부인지 알 수 없다.
`addr` 접두어를 추가하면 맥락이 좀 더 분명해진다.
`Address` 라는 클래스를 생성하면 더 좋다. 관련 데이터는 함께 묶어야 한다.

## 불필요한 맥락을 없애라
* GasStationDeluxeCustomerAccountAddress
맥락이 과도하면 오히려 읽기 어려워진다.

# 함수
프로그램의 가장 기본적인 단위가 함수다.

## 작게 만들어라
작은 함수는
* 읽기 쉽고
* 테스트하기 쉽고
* 재사용하기 쉽다.

## 더 작게 만들어라

## 블록과 들여쓰기
if / else / while 내부 블록은 가능한 단순해야 한다.
이상적인 블록은 함수 호출 한 줄 정도다.

## 한가지만 해라
함수는 반드시 하나의 역할만 수행해야 한다.

**중요한 기준**
함수 내부에서 의미 있는 이름으로 또 다른 함수를 추출할 수 있다면
현재 함수는 여러 일을 하고 있을 가능성이 크다.

## 함수당 추상화 수준은 하나로 유지하라
`getHtml()` → 매우 높은 수준의 추상화
`String pagePathName = PathParser.render(pagepath);` → 중간 수준
`.append("\n")` → 매우 낮은 수준

한 함수 내에 추상화 수준을 섞으면 코드를 읽는 사람이 헷갈린다.

## 서술적인 이름
길고 서술적인 이름이 짧고 어려운 이름보다 좋다.
* calculateMonthlySubscriptionPrice()

## 함수 인수
인수는 적을수록 좋다
이상적인 함수는 인수가 없다.

* 1개: 괜찮음
* 2개: 가능
* 3개: 신중
* 4개 이상: 특별한 이유 필요

인수가 많아질수록 이해와 테스트가 어려워진다.
웬만하면 단항 함수로 바꾸도록 애써야 한다.

## 플래그 인수
플래그 인수는 추하다.
true가 무엇을 의미하는지 알기 어렵다.
플래그 인수는 함수가 여러 역할을 수행한다는 신호이기도 하다.

나쁜 예시
```java
render(true);
```

## 인수 객체
인수가 많다면 관련 데이터를 객체로 묶어 인수를 줄일 수 있다.

## 동사와 키워드
함수 이름에 인수 이름을 넣으면 가독성이 좋아진다.

```java
writeField(name);
```

## 부수효과를 일으키지 말라
함수는 이름에 드러난 일만 해야 한다.

위험한 예시
* 전역 변수 수정
* 전달받은 객체 변경
* 세션 상태 변경

이런 동작은 예측하기 어려운 버그를 만든다.
함수가 한가지만 한다는 규칙을 위반하지만 `checkPasswordAndInitializeSession` 이라는 이름이 훨씬 좋다.

## 출력 인수를 사용하지 말라
출력 인수는 함수의 반환값(return) 대신 전달받은 인자를 직접 수정해서 결과를 만들어내는 방식을 말한다.

나쁜 예시
```java
appendFooter(report);
```
이 함수가 report에 footer를 추가하는지, footer에 report를 추가하는지 명확하지 않다.

```java
public void appendFooter(StringBuffer report);
```
선언부를 직접 확인해야 한다.

좋은 예시
```java
report.appendFooter();
```
객체 스스로 자신의 상태를 변경하도록 만드는 편이 훨씬 자연스럽고 읽기 쉽다.
