## 멀티태스킹 vs 멀티프로세싱
멀티태스킹은 운영체제가 CPU 시간을 분할하여 여러 작업을 동시에 실행되는 것처럼 보이게 하는 방식이고,
멀티프로세싱은 여러 CPU 코어를 활용해 실제로 동시에 여러 작업을 수행하는 방식이다.

* 멀티태스킹
  * 하나의 CPU 코어가 여러 작업을 번갈아 수행
  * CPU 시간을 아주 짧게 나누어 실행 (Time Slice)
  * 운영체제가 스케줄링을 통해 구현
  * 실제로는 동시에 실행되는 것이 아니라 빠르게 교체 실행
* 멀티프로세싱
  * 여러 CPU 코어를 사용하여 동시에 여러 작업을 수행
  * 진짜 병렬 처리(하드웨어 기반 동시 실행)

## 프로세스와 스레드
각 프로세스는 독립적인 메모리 공간을 갖고있다. (하나의 프로세스가 충돌해도 다른 프로세스에는 영향을 미치지 않는다)
* 코드 영역(Code) : 실행할 프로그램의 코드가 저장되는 부분
* 데이터 영역(Data) : 전역/정적 변수가 저장되는 부분
* 힙 (Heap) : 동적으로 할당되는 메모리 영역(new)
* 스택 (Stack) : 함수 호출 정보 (지역 변수와 반환 주소가 저장되는 영역)
  * 각 스레드는 자신만의 스택을 갖고있다.

스레드는 프로세스 내에서 실행되는 **작업의 단위**로 **프로세스는 하나 이상의 스레드를 반드시 포함한다.**
(프로세스는 생성될 때 기본적으로 메인 스레드 1개를 포함한다)
프로세스는 실행 환경과 자원을 제공하는 컨테이너 역할을 하고, **스레드는 CPU를 사용해서 코드를 하나하나 실행한다.**

## 컨텍스트 스위칭
스레드A를 멈추는 시점에 CPU에서 사용하던 값들을 메모리에 저장해두고
이후에 스레드A를 다시 실행할 때 이 값들을 CPU에 다시 불러와야한다.
이런 과정을 컨텍스트 스위칭이라고 한다.
컨텍스트 스위칭 과정에는 약간의 비용이 발생한다.

## CPU 바운드 작업 vs I/O 바운드 작업
각각의 스레드가 하는 작업은 크게 2가지로 구분할 수 있다.

### CPU-바운드 작업 (CPU-bound tasks)
CPU의 연산 능력을 많이 요구하는 작업을 의미한다.
이러한 작업은 주로 계산, 데이터 처리, 알고리즘 실행 등 CPU의 처리 속도가 작업 완료 시간을 결정하는 경우다.
예시: 복잡한 수학 연산, 데이터 분석, 비디오 인코딩, 과학적 시뮬레이션 등

CPU는 runnable 상태의 스레드만 실행한다.
CPU-바운드 작업에서는 스레드가 I/O로 인해 waiting 상태에 빠지지 않기 때문에 runnable 스레드가 지속적으로 존재한다.
따라서 스레드 수와 CPU 코어수가 비슷할 때 최적의 상태가 된다.

스레드 수
* CPU - 4개, 스레드 2개
  * 스레드의 숫자가 너무 적으면 모든 CPU를 100% 다 활용할 수 없다. (CPU 2개가 실행할 작업이 없어서 idle 상태로 놀게된다. 결과적으로 CPU 사용률은 100%가 아닌 50% 수준에 머물게 된다)

* CPU - 4개, 스레드 100개
  * 스레드의 숫자가 너무 많으면 CPU를 100% 다 활용할 수 있지만 불필요한 **컨텍스트 스위칭 비용**이 늘어난다.

* CPU - 4개, 스레드 4개
  * 스레드의 숫자를 CPU의 숫자에 맞춘다면 CPU를 100% 활용할 수 있고, 컨텍스트 스위칭 비용도 자주 발생하지 않기 때문에 최적의 상태가 된다. 
  * 이상적으로는 `CPU 코어 수 + 1개 정도로 스레드`를 맞추면 특정 스레드가 잠시 대기할 때 남은 스레드를 활용할 수 있다. (스레드가 잠깐 쉬는 경우가 있을 수 있다고 한다)

### I/O-바운드 작업 (I/O-bound tasks)
디스크, 네트워크, 파일 시스템 등과 같은 입출력(I/O) 작업을 많이 요구하는 작업을 의미한다.
이러한 작업은 I/O 작업이 완료될 때까지 대기 시간이 많이 발생하며 **CPU는 상대적으로 대기 상태에 있는 경우가 많다.**

쉽게 이야기해서 스레드가 CPU를 사용하지 않고 I/O 작업이 완료될 때 까지 대기한다.
예시: 데이터베이스 쿼리 처리, 파일 읽기/쓰기, 네트워크 통신, 사용자 입력 처리 등

엡 애플리케이션 백엔드 실무에서는 CPU-바운드 작업 보다는 **I/O-바운드 작업**이 많다.
CPU의 연산이 필요한 작업보다는, 대부분 사용자의 입력을 기다리거나, 데이터베이스를 호출하고 그 결과를 기다리는 등, 기다리는 일이 많다.

I/O-바운드 작업은 대부분 waiting 상태로 runnable 순간이 짧다.
따라서 스레드 수를 늘리면 CPU 활용을 증가할 수 있다.

스레드 수
* CPU 코어 4개 , 스레드 4개
  * 각 스레드는 DB 호출 후 대기 중
    * 스레드는 waiting 상태
  * CPU는 실행할 runnable 스레드가 없음

일반적인 자바 웹 애플리케이션 서버의 경우 사용자의 요청 하나를 처리하는데 1개의 스레드가 필요하기 때문에
CPU 코어에 맞추어 4개로 설정하면 동시에 4명의 사용자 요청만 처리할 수 있다.

* CPU 코어 4개 , 스레드 100개
  * 대부분의 스레드는 여전히 waiting 상태이지만 
  * 많은 스레드 중 일부는 I/O 대기에서 깨어나 실행 가능한 runnable 스레드가 되어 CPU를 사용할 수 있어 CPU 활용률이 증가한다.
