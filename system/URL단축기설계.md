`https://www.systeminterview.com/q=chatsystem&c=loggedin&v=v3&l=long` 이 입력으로 주어졌을 때
`https://tinyurl.com/y7ke-ocwj` 와 같은 단축 URL을 결과로 제공해야 한다.
단축 URL에 접속하면 원래 URL로 갈 수도 있어야 한다.
단축 URL에는 숫자(0~9), 영문자(a~z, A~Z)까지만 사용할 수 있다.

* 개략적 추정
    - 쓰기 연산: 매일 1억개의 단축 URL 생성
    - 초당 쓰기 연산: 1억(100million)/24/3600 = 1160
    - 읽기 연산: 읽기 연산과 쓰기 연산 비율은 10:1이라고 하자. 그 경우 읽기 연산은 초당 11,600회 발생한다. (1160 * 10 = 11,600)
    - URL 단축 서비스를 10년간 운영한다고 가정하면 1억(100million) * 360 * 10 = 3650억(365billion)개의 레코드를 보관해야 한다.
    - 축약 전 URL의 평균 길이는 100이라고 하면 10년 동안 필요한 저장 용량은 3650억 * 100바이트 = 36.5TB이다.

* API 설계
    * [POST] `/api/v1/data/shorten`
        * 인자: 원본 URL
        * 반환: 단축 URL

    * [GET] `/api/v1/shortUrl`
        * 반환: HTTP 리디렉션 목적지가 될 원래 URL
        * 단축 URL에 대해서 HTTP 요청이 오면 원래 URL로 보내주기 위한 용도의 엔드포인트

  예시)
    * Request URL: https://tinyurl.com/qtj5opu
    * Status Code: 301
    * Response Header
        * location: https://www.amazon.com/dp/b192FIWJ?ref=abdkw

  단축 URL을 받은 서버는 그 URL을 원래 URL로 바꾸어서 301 응답의 Location 헤더에 넣어 반환한다.

    * 301 vs 302
        * 301 Permanently Moved
            * 브라우저는 응답을 캐시 후, 추후 같은 단축 URL에 요청을 보낼 필요가 있을 때 브라우저는 캐시된 원래 URL로 요청을 보내게된다.
            * 서버의 부하를 줄일 수 있다.
        * 302 Found
            * 클라이언트의 요청은 언제나 단축 URL 서버에 먼저 보내진 후에 원래 URL로 리디렉션 되어야 한다.
            * 트래픽 분석에 용이하다.

* URL 단축
    * 해시 테이블에 <단축 URL, 원래 URL> 쌍을 저장할 수 있다.
    * 단축 URI가 `www.tinyurl.com/{hashValue}` 형태라고 가정해보자
    * 긴 URL을 해시 함수에 넣은 해시 결과를 단축 URL에 사용할 수 있다.

* 데이터 모델
  메모리는 유한하고 비싸기 때문에 관계형 데이터베이스에 저장하는 것이 좋다.

    * id(pk)
    * shortURL
    * longURL

* 해시 함수
  해시 함수는 원래 URL을 단축 URL로 변환하는데 쓰인다.
  hashValue는 [0-9,a-z,A-Z]의 문자들로 구성되므로 사용할 수 있는 문자의 개수는 10 + 26 + 26 = 62개이다.
  개략적 추정치에 따르면 3650억 개의 URL을 만들어 낼 수 있어야 한다.

| hashValue의 길이 | 해시 함수가 만들 수 있는 URL의 개수     |
|---------------|----------------------------|
| 1             | 62^1 = 61                  |
| 2             | 62^2 = 3,844               |
| 3             | 62^3 = 238,328             |
| 4             | 62^4 = 14,776,336          |
| 5             | 62^5 = 916,132,832         |
| 6             | 62^6 = 56,800,235,584      |
| 7             | 62^7 = 3,521,614,606,208   |
| 8             | 62^8 = 218,340,105,584,896 |

n=7이면 3.5조개의 URL을 만들 수 있다.
이는 요구사항을 만족시키기 충분한 값이므로 hashValue의 길이를 7로 하겠다.

* 해시 후 충돌 해소

  긴 URL을 줄이려면 원래 URL을 7글자 문자열로 줄이는 해시 함수 필요하다.
  CRC32, MD5, SHA-1와 같이 잘 알려진 해시 함수를 이용할 수 있다.
  위 해시 함수들을 사용하여 `https://en.wikipedia.org/wiki/Systems_design`을 축약한 결과다.

| 해시함수  | 해시 결과 (16진수)                             |
|-------|------------------------------------------|
| CRC32 | 5cb54054                                 |
| MD5   | 5a62509a84df9ee03fe1230b9df8b84e         |
| SHA-1 | 0eeae7916c06853901d9ccbefbfcaf4de57ed85b |

CRC32가 계산한 가장 짧은 해시값조차도 7보다는 길다.
계산된 해시 값에서 처음 7개 글자만 이용할 수 있지만 이 경우 해시 결과가 서로 충돌할 확률이 높아진다.

예시)

```
https://example.com/abc  → 해시 →  a1b2c3d
https://example.com/xyz  → 해시 →  a1b2c3d   ❌ 충돌 발생
```

두 URL이 동일한 해시 앞 7글자를 가지면 충돌이 발생한다.
이를 해결하기 위해 사전에 정한 원래 URL에 덧붙여 다시 해시를 수행한다.

1. 충돌을 피하기 위해 "xyz" 뒤에 +1 문자열 추가해서 다시 해시

```
해시("https://example.com/xyz" + "1") → 3c8d0e2
```

2. 그래도 충돌나면 "2" 붙여서 다시 해시

```
해시("https://example.com/xyz" + "2") → 98d12ff
```

이 과정을 충돌이 없어질 때까지 반복한다.

* base-62 변환
    * 10진수로 11157, 즉 11157을 62진수로 변환해보겠다.
    * 62진법은 수를 표현하기 위해 총 62개의 문자를 사용하는 진법이다.
        * 0은 0으로, 9는 9로, 10은 a로, 11은 b로, 35는 z로, 36은 A로, 61은 Z로 대응한다.
    * 11157(10)은 [2, 55, 59] → [2, T, X] → 2TX(16)이다.
    * 따라서 단축 URL은 https://tinyurl.com/2TX가 된다.

두 접근법 비교

| 해시 후 충돌 해소 전략                                          | base-62 변환                                                                  |
|--------------------------------------------------------|-----------------------------------------------------------------------------|
| 단축 URL의 길이가 고정됨                                        | 단축 URL의 길이가 가변적. ID 값이 커지면 같이 길어짐                                           |
| 유일성이 보장되는 id 생성기가 필요하지 않음                              | 유일성이 보장되는 id 생성기가 필요                                                        |
| 충돌이 가능해서 해소 전략 필요                                      | id의 유일성이 보장된 후에야 적용 가능한 전략이라 충돌은 아예 불가능                                     |
| id로부터 단축 URL을 계산하는 방식이 아니라 다음에 쓸 수 있는 URL을 알아내는 것이 불가능 | id가 1씩 증가하는 값이라고 가정하면 다음에 쓸 수 있는 단축 URL이 무엇인지 쉽게 알아낼 수 있어서 보안상 문제가 될 소지가 있음 |

## URL 단축기 상세 설계
1. 입력으로 긴 URL을 받는다.
2. 데이터베이스에 해당 URL이 있는지 검사한다.
3. 데이터베이스에 있다면 해당 URL에 대한 단축 URL을 만든적이 있는 것이므로 클라이언트에게 바로 반환
4. 데이터베이스에 없다면 유일한 ID(pk)를 생성한다.
   1. 입력된 URL이 https://en.wikipedia.org/wiki/System_design 이라고 하자. 
   2. 이 URL에 대해 id 생성기가 반환한 id는 2009215674938 이다.
5. 62진법 변환을 적용하여 ID를 단축 URL로 만든다.
   3. 이 id를 62진수로 변환하면 zn9edcu를 얻는다.
6. ID, 단축 URL, 원래 URL로 데이터베이스 레코드를 만든 후 단축 URL을 클라이언트에 전달한다.
   * id: 2009215674938
   * shortUrl: zn93dcu
   * longUrl: https://en.wikipedia.org/wiki/System_design

## URL 리디렉션 상세 설계
쓰기보다 읽기를 더 자주 하는 시스템이기 때문에 <단축 URL, 원래 URL> 의 쌍을 캐시에 저장하여 성능을 높일 수 있다.

1. 사용자가 단축 URL을 클릭한다.
2. 로드밸런서가 해당 클릭으로 발생한 요청을 웹 서버에 전달한다.
3. 단축 URL이 이미 캐시에 있는 경우 원래 URL을 바로 꺼내서 클라이언트에게 전달한다.
4. 캐시에 해당 단축 URL이 없는 경우에는 데이터베이스에서 꺼낸다.
5. 데이터베이스에서 꺼낸 URL을 캐시에 넣은 후 사용자에게 반환한다.
