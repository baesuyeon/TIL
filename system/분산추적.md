## 분산 추적(Distributed Tracing)
여러 마이크로서비스 간의 호출 경로를 시각화하고 성능 병목 지점을 찾는 기술
관련 시스템: 구글의 Dapper, 트위터의 Zipkin, 우버의 Jaeger, Datadog

### Trace, Span
Trace와 Span은 Datadog을 포함한 모든 분산 추적 시스템의 표준 용어이다.
Trace(트레이스): 클라이언트 요청이 들어와서 발생하는 전체 호출 경로를 의미하며 이를 DAG(비순환 방향 그래프) 형태로 표현한다.
Span(스팬): 트레이스 내부의 개별 서비스 호출 단위를 의미한다. 예를 들어 서비스 A가 B, C, D를 호출할 때 각각의 호출이 하나의 스팬이 된다.
데이터베이스 호출이나 메세지 브로커를 통한 비동기 통신도 스팬으로 기록할 수 있다.

## 시스템 요구사항
* 추적 시스템이 실제 서비스의 처리 성능에 영향을 주어서는 안된다.
* 엔지니어가 각 스팬의 시작 및 종료 시간을 확인하고 전체 트레이스를 볼 수 있는 UI가 필요하다.
* 추적 데이터가 웹 인터페이스에 나타나기까지 몇 분 정도의 지연은 허용된다.
* 수집된 추적 데이터는 1년 동안 보관하는 것을 목표로 한다.

### Sampling
모든 요청을 기록하는 것은 저장 비용 및 시스템에 너무 큰 부하를 주기 때문에 다운 샘플링(Down-sampling)이 필수적이다.
샘플링은 시스템이 합리적인 시간 내에 데이터를 처리할 수 있도록 돕는다.
Uber의 Jaeger 시스템의 경우 기본적으로 0.1%의 샘플링 비율(1,000개의 쿼리 중 1개만 기록)을 사용하도록 설계되었다.

예시를 통해 시스템이 처리해야할 데이터 양 계산해보기
기본 환경: 1,000개의 내부 서비스, 서비스당 평균 100대의 서버, 서버당 초당 100개의 쿼리(QPS)를 처리한다고 가정
전체 트래픽: 전체 시스템은 초당 약 1,000만 QPS(1,000 서비스 × 100 머신 × 100 QPS)를 처리
수집 데이터양: 0.1% 샘플링을 적용하면, 시스템은 초당 약 10,000개의 스팬을 기록

* 1시간: 3,600초 (60 * 60)
* 1일: 24시간 * 3600초 = 86,400초
하루에 10,000 * 약 100,000 = 1,000,000,000개, 약 10억개 (1 billion)
1년에 1,000,000,000 × 365 = 365,000,000,000, 약 3650억개

## 데이터베이스
분산 추적 시스템은 초당 약 10,000개의 쓰기 작업이 발생하는 쓰기 집약적(Write-heavy) 시스템이다.
0.1% 샘플링을 적용하더라도 1년에 약 3,000억 개의 스팬(Span) 데이터가 생성된다.
수평 확장이 쉽고 대용량 처리에 최적화된 DB를 사용해야한다.

Cassandra
* 쓰기가 굉장히 빠르다 (write-heavy에 최적)
* 샤딩 기본 지원 (자동 분산)
카산드라는 대량의 쓰기 작업에 최적화되어있으며 수평적 확장이 용이하다.

## 스키마 설계
Partition Key는 데이터를 어느 서버에 저장할지 결정하는 기준 값이 된다.
Partition Key를 잘못 정하면 특정 서버에만 데이터가 몰리는 이슈가 발생할 수 있다.

Partition Key = Trace ID
```text
Trace A
 ├ span1
 ├ span2
 ├ span3
```
같은 요청(trace)의 span들을 한곳에 모은다.
