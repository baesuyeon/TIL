# 대용량 트래픽 기차 예매 시스템 아키텍처
## 문제 정의
기차 예매 시스템은 예약 오픈 시점에 맞춰 짧은 시간 내 매우 많은 요청이 몰린다.
이때 만족해야 하는 조건은 아래와 같다.
* 좌석 수를 초과하는 예매가 발생하면 안된다.
* 각 좌석은 중복 예약이 발생해서는 안된다.
* 사용자에게 현재 대기 순번을 보여줘야 한다.
* 선착순 순서가 어느 정도 보장되어야 한다.

## 대기열 등록
```text
Client → API → DB
```
위와 같이 가장 단순한 구조에서
수십만 요청이 한번에 들어오면 API 서버는 늘릴 수 있어도 DB는 병목이 된다.

따라서 모든 요청을 DB로 보내지 않고
먼저 인메모리 대기열에 적재해 트래픽을 흡수해야 한다.

대기열 자료구조: Redis Sorted Set
사용자가 예약 시작 버튼을 누르면 API는 UUID를 발급하고 이를 Redis Sorted Set에 넣는다.

```text
ZADD queue:{trainId}:{scheduleId} <timestamp> <uuid>
```
* member = uuid
* score = timestamp
Sorted Set은 score 기준으로 정렬되므로 대기열 순서를 관리하기 좋다.

userId를 member로 사용할 수도 있지만 여러 디바이스 접속이나 재접속 복구 문제 때문에
보통 queueToken(UUID)을 발급해서 세션 단위로 대기열을 관리한다.

같은 사용자가 같은 UUID로 다시 요청하면
ZSET의 같은 member는 중복 삽입되지 않고 score만 갱신된다.
다만 같은 member일때만 ZSET이 중복을 방지해줄 수 있기 때문에
서버에서 1회 발급한 UUID를 클라이언트가 계속 재사용해야한다.

## 대기 순번 조회
대기열 시스템에서 사용자는 자신의 상태를 계속 확인하고 싶어한다.
Redis Sorted Set은 이를 빠르게 지원한다.

내 순번 조회
```text
ZRANK queue:{trainId}:{scheduleId} <member>
```

전체 대기열 길이
```text
ZCARD queue:{trainId}:{scheduleId}
```
MQ도 요청 대기열을 순서대로 소비하는 것은 가능하지만
내가 몇번째인지 지금 전체 대기 인원이 몇 명인지 
재접속해도 같은 위치를 유지하는지 (Redis는 NX로 member가 없을 때만 추가가 가능, ZADD queue NX timestamp uuid)와 같은
요구사항을 만족시키기 어렵다.

## Active User로 이동
대기열에서 일정 수의 사용자만 실제 예매 단계로 보내야 한다.
이 역할을 스케줄러 또는 워커가 수행한다.

```text
ZPOPMIN queue:{trainId}:{scheduleId} 1000
```
가장 앞의 1000명을 꺼내기

```text
SET active:{uuid} true EX(TTL) 300
```
Active User로 등록하기
가장 앞선 1000명을 꺼내서 Active User로 등록한다. 
+) TTL 설정하여 유휴 유저는 자동 제거되도록한다.

누구나 예매 API에 접근 가능하면 결국 Booking API와 DB가 터질 수 있다.
Queue는 줄 세우기, Active User는 입장권 역할을 한다.

사용자는 polling 으로 상태를 확인하다가
Active 상태가 되면 그때만 실제 예매 화면으로 진입한다.

## 특정 좌석 중복 예매 방지
대기열을 통과했다고 해서 예약이 보장되는 것은 아니다.
여러명이 동시에 같은 시간대, 같은 좌석을 선택할 수 있기 때문이다.

### 총 수량 초과 방지
기차 예매가 아닌, 스탠딩 공연과 같이 좌석이 랜덤 배정인 경우 남은 수량만 관리하면 된다.
남은 좌석이 10석이면 11번째 요청은 실패해야 한다.
