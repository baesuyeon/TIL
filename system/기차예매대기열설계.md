# 대용량 트래픽 기차 예매 시스템 아키텍처
## 문제 정의
기차 예매 시스템은 예약 오픈 시점에 맞춰 짧은 시간 내 매우 많은 요청이 몰린다.
이때 만족해야 하는 조건은 아래와 같다.
* 좌석 수를 초과하는 예매가 발생하면 안된다.
* 각 좌석은 중복 예약이 발생해서는 안된다.
* 사용자에게 현재 대기 순번을 보여줘야 한다.
* 선착순 순서가 어느 정도 보장되어야 한다.

## 전체 흐름
* 사용자 대기열 등록
* 사용자 polling
* Worker가 N명 Active 전환
* Client가 Active 감지
* 예약 화면 진입
* 좌석 선택
* 예약 API 호출 (POST /booking)
* Token signature 검증
* Token expire 검증
* trainId/scheduleId 검증
* Redis active 검증
* Redis Lock
* DB Unique Constraint

## 대기열 등록
```text
Client → API → DB
```
위와 같이 가장 단순한 구조에서
수십만 요청이 한번에 들어오면 API 서버는 늘릴 수 있어도 DB는 병목이 된다.

따라서 모든 요청을 DB로 보내지 않고
먼저 인메모리 대기열에 적재해 트래픽을 흡수해야 한다.

대기열 자료구조: Redis Sorted Set
사용자가 예약 시작 버튼을 누르면 API는 UUID를 발급하고 이를 Redis Sorted Set에 넣는다.

```text
ZADD queue:{trainId}:{scheduleId} <timestamp> <uuid>
```
* member = uuid
* score = timestamp
Sorted Set은 score 기준으로 정렬되므로 대기열 순서를 관리하기 좋다.

userId를 member로 사용할 수도 있지만 여러 디바이스 접속이나 재접속 복구 문제 때문에
보통 queueToken(UUID)을 발급해서 세션 단위로 대기열을 관리한다.

같은 사용자가 같은 UUID로 다시 요청하면
ZSET의 같은 member는 중복 삽입되지 않고 score만 갱신된다.
다만 같은 member일때만 ZSET이 중복을 방지해줄 수 있기 때문에
서버에서 1회 발급한 UUID를 클라이언트가 계속 재사용해야한다.

### userId를 사용하지 않는 이유
대기열 시스템은 사용자 계정이 아니라 접속 세션 단위로 관리되어야 한다.
userId를 member로 사용할 경우 다음 문제가 발생할 수 있다.

- 동일 사용자의 멀티 디바이스 접속 시 대기열 충돌
- 새로고침 및 재접속 시 대기열 위치 복구 어려움
- 로그인 이전 단계에서 대기열 관리 불가

따라서 대기열 시스템에서는 queueToken(UUID) 을 발급하여 세션 단위로 대기열을 관리하는 것이 일반적이다

## 대기 순번 조회
대기열 시스템에서 사용자는 자신의 상태를 계속 확인하고 싶어한다.
Redis Sorted Set은 이를 빠르게 지원한다.

내 순번 조회
```text
ZRANK queue:{trainId}:{scheduleId} <member>
```

전체 대기열 길이
```text
ZCARD queue:{trainId}:{scheduleId}
```
MQ도 요청 대기열을 순서대로 소비하는 것은 가능하지만
내가 몇번째인지 지금 전체 대기 인원이 몇 명인지 
재접속해도 같은 위치를 유지하는지 (Redis는 NX로 member가 없을 때만 추가가 가능, ZADD queue NX timestamp uuid)와 같은
요구사항을 만족시키기 어렵다.

## Active User로 이동
대기열에서 일정 수의 사용자만 실제 예매 단계로 보내야 한다.
이 역할을 스케줄러 또는 워커가 수행한다.

```text
ZPOPMIN queue:{trainId}:{scheduleId} 50
```
1초마다 가장 앞의 50명을 꺼내기

```text
SET active:{trainId}:{scheduleId}:{uuid} true EX(TTL) 300
```
Active User로 등록하기
가장 앞선 50명을 꺼내서 Active User로 등록한다. 
+) TTL 설정하여 유휴 유저는 자동 제거되도록한다.

누구나 예매 API에 접근 가능하면 결국 Booking API와 DB가 터질 수 있다.
Queue는 줄 세우기, Active User는 입장권 역할을 한다.

* 사용자는 polling 으로 상태를 확인하다가
  * GET /queue/status?token={서버 발급 토큰} 호출
* Active 상태가 되면 그때만 실제 예매 화면으로 진입한다.
  * 서버는 Redis 조회 (GET active:xxx)
  * false → 아직 대기
  * true → 입장 가능 → redirect("/reservation")

## 특정 좌석 중복 예매 방지
대기열을 통과했다고 해서 예약이 보장되는 것은 아니다.
여러명이 동시에 같은 시간대, 같은 좌석을 선택할 수 있기 때문이다.

예를 들어 아래와 같은 상황이 발생할 수 있다.
```text
User A → 좌석 12A 선택
User B → 좌석 12A 선택
```

두 요청이 동시에 서버에 도착하면 다음과 같은 문제가 발생할 수 있다.
```text
1. 두 요청 모두 좌석이 비어있다고 판단
2. 두 요청 모두 예약 시도
3. 동일 좌석이 중복 예약
```
이를 방지하기 위해 좌석 예약 시 동시성 제어(Concurrency Control) 가 필요하다.

**방법 1: DB Unique Constraint (가장 기본적인 방법)**

좌석 테이블에 유니크 제약 조건을 둔다.
```text
UNIQUE(train_id, schedule_id, seat_no)
```
예약 요청이 동시에 들어오더라도 `INSERT booking(...)` 시점에 `duplicate key error`  가 발생하여 하나의 요청만 성공하게 된다.
이 방법은 **가장 단순하면서도 안정적인 방식**이다.

**방법 2: Redis 분산 락**

좌석 단위로 Redis Lock을 잡아 동시 요청을 직렬화할 수 있다.
```text
SET lock:seat:{trainId}:{scheduleId}:{seatNo} 1 NX EX 5
```
NX → 이미 lock이 있으면 실패
EX → 일정 시간 후 자동 해제

```text
좌석 선택
   ↓
Lock 획득 시도
   ↓
성공 → 예약 진행
실패 → 이미 예약 중
```
이 방식은 DB 부하를 줄이는 장점이 있다.

방법 3: Optimistic Lock (버전 기반)

좌석 테이블에 version 컬럼을 두고 업데이트 시 검증할 수 있다.
```text
seat
-----
seat_no
status
version
```
```text
UPDATE seat
SET status = 'BOOKED', version = version + 1
WHERE seat_no = '12A'
AND status = 'AVAILABLE'
AND version = 3
```
다른 요청이 먼저 예약하면 update row = 0  이 되어 예약 실패 처리된다.

실무에서는 보통 아래 조합을 사용한다.
```text
1차 방어 → Redis Lock
2차 방어 → DB Unique Constraint
```
대규모 트래픽 환경에서는 **충돌을 DB까지 보내는 것 자체가 비용**이기 때문에
실무에서는 분산락을 더 많이 사용한다. (DB write 부하가 폭증하는 문제 방지)

## 최종 예약 저장

### 총 수량 초과 방지
기차 예매가 아닌, 스탠딩 공연과 같이 좌석이 랜덤 배정인 경우 남은 수량만 관리하면 된다.
남은 좌석이 10석이면 11번째 요청은 실패해야 한다.

## API를 직접 호출하는 케이스 방어
사용자가 Queue / Active 과정을 우회하고 Booking API를 직접 호출하는 문제가 있을 수 있다.

* Active Token을 검증한다
  * SET active:{서버 발급 토큰} true EX 300
  * 서버에서 Redis active token을 확인하고 없다면 차단한다.
  * 5분 TTL이 걸려있어 token을 탈취하더라도 오래 사용하지 못한다.

### 토큰
```text
POST /booking
queueToken = random uuid
```
queueToken(UUID)만 쓰면 추측 공격 가능성이 있다.
토큰이 서버가 발급했다는 보장이 없기 떄문에 서버에서는 검증할 방법이 없다.

```text
payload = uuid + ":" + trainId + ":" + scheduleId + ":" + timestamp
String signature = HmacSHA256(payload, SERVER_SECRET);

token = payload + "." + signature;
```
```text
String[] parts = token.split(".");
payload = parts[0]
signature = parts[1]

expected = HmacSHA256(payload, SECRET)

if (!signature.equals(expected)) {
    throw InvalidToken
}
```
생성된 토큰: abc123:trainA:10:00:1710000000.signature
임의로 생성한 token을 검증할 수 있다.

+) uuid만 사용한다면 토큰이 특정 예매에 묶여 있지 않아 (어떤 열차, 어떤 스케쥴에 대한 정보가 없어)
열차 A의 대기열을 통과해 발급받은 토큰을 이용해 열차 B의 예약을 시도하는 것이 가능해진다.
따라서 토큰 payload에 예약 대상 리소스를 넣어야 한다.
또한 payload에 대한 서명(signature) 을 추가하여 클라이언트가 payload를 임의로 조작하지 못하도록 한다.
