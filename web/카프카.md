## 에러 처리
Producer가 메시지를 전송한 이후 Kafka 토픽에 적재된 메시지를 Consumer가 처리하는 과정에서 다양한 오류가 발생할 수 있다.
예를들어 Consumer 애플리케이션이 일시적으로 중단되거나 외부 시스템(DB, API 등)과의 연결이 일시적으로 불가능한 상황이 발생할 수 있다.
이때 적절한 에러 핸들링 전략이 없다면 메시지 유실과 같은 문제가 발생할 수 있다.

## Spring Boot 기본 에러 핸들링
Spring Boot의 기본 에러 핸들링은 `DefaultErrorHandler` 인데 기본으로 설정되며
`Dead Letter Topic`도 `Backoff` 재시도도 없다. 동작방식은 아래와 같다.
* 실패 시 최대 10번 재시도
  * 연속으로 9번 재시도(0ms) 후에도 실패하면 해당 레코드를 로그만 남기고 넘어간다 (skip)
  * 즉 메시지가 유실된다. (다음 레코드를 진행하고 정상적으로 offset이 커밋된다)

## 커스텀 에러 핸들링
```kotlin
@Bean
fun errorHandler(template: kafkaTemplate<String, String>): DefaultErrorHandler {
    val recoverer = DeadLetterPublishingRecoverer(template) // DLT로 보내기
    
    val backOff = ExponentialBackOff(
        1000L, // 첫 재시도 대기 시간(1초)
        2.0    // multiplier: 재시도마다 대기 시간을 2배로 증가(1s -> 2s -> 4s -> 8s)
    ).apply {
        maxElapsedTime = 60000L // 재시도 시작 후 총 경과 시간이 60초를 넘으면 재시도 중단
    }
    
    return DefaultErrorHandler(recoverer, backoff)
}
```
exponential backoff로 재시도하다가 최종 실패 시 `{원래토픽}.DLT` 토픽으로 메시지를 보낸다.
실무에서는 거의 반드시 커스텀 `ErrorHandler`를 등록해야 한다.

## 역직렬화 에러
`poll()` 단계에서 바이너리(Byte) 데이터를 객체(Object)로 변환하는 과정 중 에러가 발생할 수 있다.
(스키마가 불일치하거나, 데이터를 꺼낼 때 데이터가 어떤 형태인지 확인하기 위해 설명서(Schema Registry)를 참조해야하는데 네트워크 문제로 잠깐 못 읽는 상황이 발생할 수 있다)

`DefaultErrorHandler`는 이 역직렬화 에러를 처리하지 못한다.
1. poll: 카프카에서 데이터를 가져온다.
2. deserialize: 데이터를 객체로 변환한다.
3. invoke: 비즈니스 로직을 실행한다.
4. error handling: 비즈니스 로직 실행 중 에러가 나면 처리한다.
위 과정에서 역직렬화에 실패하면 데이터가 객체로 변환이 되지 않아 3번으로 넘어가지 못한다. 

1. `poll()` → 레코드 역직렬화 실패 → 예외 발생
2. 해당 레코드의 offset이 커밋되지 않음
3. 다음 `poll()`에서 같은 레코드를 다시 가져옴
4. 역직렬화 실패 → 1번 과정으로 돌아감
만약 바이트 자체가 잘못된 경우 몇 번을 시도해도 절대 성공할 수 없게된다. 
(무한 재시도하며 컨슈머는 나 이거 못읽어! 하고 뻗어버린다)

Spring Kafka는 해결책으로 `ErrorHandlingDeserializer`를 전달하는데
`ErrorHandlingDeserializer`는 에러가 발생하더라도 예외를 던지지 않고
에러 내용을 담은 가짜 객체를 만들어서 4번까지 전달하는 역할을 한다.
그러면 `DefaultErrorHandler`가 개입할 수 있게되고 skip이나 DLT 발행같은 처리가 가능해진다.

```text
spring:
  kafka:
    consumer:
      key-deserializer: org.springframework.kafka.support.serializer.ErrorHandlingDeserializer
      # 실제 데이터를 읽는 도구를 ErrorHandlingDeserializer로 감싸겠다고 선언
      value-deserializer: org.springframework.kafka.support.serializer.ErrorHandlingDeserializer
      properties:
        spring.deserializer.key.delegate.class: org.apache.kafka.common.serialization.StringDeserializer
        # 실제 역직렬화에 사용할 클래스는 여기에 별도로 지정
        spring.deserializer.value.delegate.class: org.springframework.kafka.support.serializer.JsonDeserializer
```
설정 파일(application.yml)에 명시를 해줘야 스프링이 ErrorHandlingDeserializer로 실제 deserializer를 감싼다.
