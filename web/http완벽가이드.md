# 1장 HTTP 개관

HTTP는 웹에서 클라이언트와 서버가 데이터를 주고받기 위해 사용하는 애플리케이션 계층 프로토콜이다.
HTTP는 신뢰성 있는 전송 프로토콜(TCP) 위에서 동작하기 때문에 데이터가 전송 중

* 손상되거나
* 순서가 바뀌거나
* 중복 전달되는 문제를 걱정할 필요가 없다.

HTTP는 웹의 멀티미디어 배달부 역할을 하며 다양한 데이터 타입을 전달할 수 있다.

## 웹 리소스

웹 서버는 웹 리소스를 관리하고 제공한다.
리소스는 크게 두가지로 나뉜다.

**정적 리소스**
서버에 저장된 파일을 그대로 전달한다.

* 텍스트 파일
* HTML 파일
* JPEG 이미지 파일
* AVI 동영상 파일

**동적 리소스**
서버 프로그램이 요청에 따라 동적으로 생성하는 콘텐츠

* 검색 결과 페이지
* 로그인 사용자별 페이지
* API 응답 JSON

## MIME 타입

인터넷에서는 수천 가지 데이터 타입이 존재하기 때문에 HTTP는 MIME 타입을 사용해 데이터 타입을 명시한다.
서버는 응답 메시지의 `Content-Type` 헤더에 데이터 타입을 포함한다.

`Content-type: image/jpeg`
브라우저는 이 MIME 타입을 보고 데이터를 어떻게 처리할지 결정한다.

대표적인 MIME 타입

| 타입               | 설명       |
|------------------|----------|
| text/html        | HTML 문서  |
| text/plain       | 일반 텍스트   |
| image/jpeg       | JPEG 이미지 |
| image/gif        | GIF 이미지  |
| application/json | JSON 데이터 |

## URL

URL은 웹에서 리소스의 위치와 접근 방법을 나타내는 주소이다.

기본 구조
`scheme://host:port/path?query#fragment`

예
`http://www.example.com:80/index.html?name=bob#section1`

| 구성요소     | 설명                                             |
|----------|------------------------------------------------|
| scheme   | 리소스에 접근하기 위해 사용되는 프로토콜, 보통 HTTP 프로토콜 (http://) |
| host     | 서버 주소                                          |
| port     | 서버 포트                                          |
| path     | 서버 리소스 위치                                      |
| query    | 추가 요청 파라미터                                     |
| fragment | 문서 내부 위치                                       |

## HTTP 트랜잭션
HTTP 통신은 요청(Request) 과 응답(Response) 으로 이루어진다.
이를 HTTP 트랜잭션이라 한다.

요청 메시지
```text
GET /test/hi-there.txt HTTP/1.0
Accept: text/*
Accept-Language: en, fr
```

응답 메시지
```text
HTTP/1.0 200 OK
Content-Type: text/plain
Content-length: 19

Hi! I'm a message!
```

### HTTP 메시지 구조
HTTP 메시지는 다음과 같은 구조를 가진다.
```text
<시작줄>
<헤더>
<빈 줄>
<본문>
```

**시작줄**
요청
```text
GET /index.html HTTP/1.1
```

응답
```text
HTTP/1.1 200 OK
```

**헤더**
헤더는 메시지의 메타데이터를 제공한다.
시작 줄 이후에 0개 이상의 헤더 필드가 이어진다.
`:`로 구분되는 이름-값으로 구성된다
헤더는 빈 줄로 끝난다.
```text
Content-Type: text/html
Content-Length: 100
```

**본문**
본문은 실제 전달되는 데이터이다.
* HTML
* JSON
* 이미지
* 파일

## TCP 커넥션
```text
HTTP                    - 애플리케이션 계층
TCP                     - 전송 계층
IP                      - 네트워크 계층
네트워크를 위한 링크 인터페이스 - 데이터 링크 계층
물리적인 네트워크 하드웨어     - 물리 계층
```

HTTP는 자신의 메시지 데이터를 전송하기 위해 TCP를 사용한다.
위 메시지가 TCP 커넥션을 통해 어떻게 전송되는지 알아보자.

TCP는 다음을 제공한다.
* 오류없는 데이터 전송
* 순서에 맞는 전달 (데이터는 언제나 보낸 순서대로 도착한다)
* 조각나지 않는 데이터 스트림

TCP/IP는 어떤 종류의 컴퓨터든 네트워크든 서로 신뢰성있는 의사소통을 하게해준다.

클라이언트가 서버에 메시지를 전송하기 위해 IP 주소와 포트번호를 사용해 클라이언트와 서버 사이의 TCP/IP 커넥션을 맺어야 한다.
IP 주소와 포트 번호는 URL을 통해 알아낼 수 있다.

## HTTP 요청 과정
브라우저가 웹 페이지를 요청하는 과정

* 웹브라우저는 서버의 URL에서 호스트 명을 추출한다.
* 웹브라우저는 DNS를 조회하여 서버의 호스트 명을 IP로 변환한다.
* 웹브라우저는 URL에서 포트번호(없다면 80)를 추출한다.
* 웹브라우저는 웹 서버와 TCP 커넥션을 맺는다. (TCP 3-way handshake)
* 웹브라우저는 서버에 HTTP 요청을 보낸다.
* 서버는 웹브라우저에 HTTP 응답을 돌려준다.
* 커넥션이 닫히면 웹브라우저 문서를 보여준다.

## 프록시
```text
Client → Proxy → Server
```
프록시 서버는 클라이언트와 서버 사이에 위치한 중개 서버이다.

프록시 역할
* 보안
* 캐싱
* 로깅
* 접근 제어

웹 캐시는 특수한 프록시 서버이다.
캐시에 리소스가 존재하면 원 서버에 요청하지 않고 캐시에서 응답한다.

장점
* 응답 속도 향상
* 네트워크 트래픽 감소
* 서버 부하 감소

# 2장 URL과 리소스
URL은 리소스의 위치 + 접근 방법을 나타낸다.

예
```text
http://www.example.com:80/index.html?name=bob#top
```

* 스킴: 사용할 프로토콜
    * 스킴은 리소스에 어떻게 접근할지 알려준다.
    * http, https, ftp, mailto, file
* 호스트와 포트

| 프로토콜  | 기본 포트 |
| ----- |-------|
| HTTP  | 80    |
| HTTPS | 443   |
| FTP   | 21    |

* 경로
    * 리소스의 서버 내 위치
* 질의 문자열
    * 서버에 전달되는 추가 파라미터 (key=value)
    * ?name=bob&age=20
* 프래그먼트
    * 문서 내부 특정 위치를 가리킨다.
    * 브라우저 내부에서만 사용된다.

## 상대 URL

상대 URL은 기저 URL(Base URL) 을 기준으로 해석된다.

기저 URL
```text
http://www.joes-hardware.com/tools.html
```

상대 URL
```text
./hammers.html
```

결과 
```text
http://www.joes-hardware.com/hammers.html
```

기저 URL을 사용하여 상대 URL에서 기술하지 않은 정보를 추측할 수 있다.
* 상대 URL이 `./hammers.html`, 기저 URL이 `http://www.joes-hardward.com/tools.html` 인 경우
* 기저 URL의 스킴(http), 호스트(www.joes-hardward.com), 포트(80)를 상속받는다.
* 상대 컴포넌트와 상속받은 컴포넌트를 합쳐 새로운 절대 URL `http://www.joes-hardward.com/hammers.html` 을 만든다.

## URL 인코딩
URL에서는 특정 문자를 사용할 수 없다.
예
* 공백
* 한글
* 특수문자
이러한 문자는 % + 16진수 ASCII 코드로 변환된다.

| 문자    | 인코딩         |
| ----- | ----------- |
| space | %20         |
| !     | %21         |
| 한글    | UTF-8 후 인코딩 |

`hello world` → `hello%20world`

## URI vs URL
URI > URL

| 개념  | 설명              |
| --- | --------------- |
| URI | 리소스를 식별하는 모든 방법 |
| URL | 리소스 위치          |
| URN | 리소스 이름          |

URL: https://example.com/index.html
URN: urn:isbn:9780131101630

URL은 리소스를 찾는데 필요한 서버 이름과 포트를 제공한다.
리소스가 옮겨지면 URL을 더이상 사용할 수 없다는 단점이 있다.

## PURL
URL의 문제점은 리소스가 이동하면 URL이 깨진다는 것이다.
이를 해결하기 위해 PURL(Persistent URL) 이 등장했다.

```text
Client → PURL 서버 → 실제 URL
```
PURL 서버가 리소스의 실제 위치를 관리한다.
따라서 리소스 위치가 바뀌어도 PURL은 변하지 않는다.
클라이언트는 위치 할당자에게 리소스를 가져올 수 있는 영구적인 URL을 요청할 수 있다.

# 3장 HTTP 메시지
* 메시지는 인바운드로 이동하고(서버 쪽으로), 아웃바운드로 이동한다(클라이언트 쪽으로)
* 메시지는 다운스트림으로만 흐른다.
* 메시지는 시작줄, 헤더, 본문(body)로 이루어진다.
  * 시작줄: HTTP 요청 메서드, 요청 대상, HTTP Version
  * 헤더: HTTP 전송에 필요한 모든 부가정보
    * 표준 헤더가 존재하며 필요시 임의의 헤더를 추가가능하다.
  * 본문은 데이터를 담고있다. (Optional)
    * HTML 문서, 이미지, 영상, JSON 등등 byte로 표현할 수 있는 모든 데이터 전송 가능
    * 본문이 없더라도 CRLF(줄바꿈, `\r\n`)로 끝나게된다.
* 요청 메시지는 웹 서버에 어떤 동작을 요구한다.
* 응답 메시지는 요청의 결과를 클라이언트에 돌려준다.

### 요청 메시지
```text
<메서드> <요청 URL> <버전>
<헤더>

<본문>
```
메서드 

### 응답 메시지
```text
<버전> <상태 코드> <사유 구절>
<헤더>

<본문>
```

API를 설계할 때, 리소스와 행위를 분리할 수 있다.

## 메서드(행위)
* GET : 리소스 조회
  * 호출해도 리소스를 변경하지 않아 안전하다.
* POST : 요청 데이터 처리, 주로 등록에 사용
* PUT : 리소스를 대체, 해당 리소스가 없으면 생성(파일을 폴더에 PUT 하는 것 처럼)
* PATCH : 리소스 부분 변경
* DELETE : 리소스 삭제

* HEAD
  * GET과 동일하지만 본문(body)을 반환하지 않는 요청
  * 리소스 존재 여부나 Content-Length, 캐시 상태 등을 확인하기 위해 사용한다.

요청 예시
```text
HEAD /index.html HTTP/1.1
Host: example.com
```

응답 예시
```text
HTTP/1.1 200 OK
Date: Fri, 13 Mar 2026 13:00:00 GMT
Content-Type: text/html
Content-Length: 1256
Last-Modified: Mon, 10 Mar 2026 10:00:00 GMT
Server: nginx
```

* OPTIONS
  * 서버가 지원하는 HTTP 메서드 확인

요청 예시
```text
OPTIONS /users HTTP/1.1
Host: example.com
```

응답 예시
```text
HTTP/1.1 204 No Content
Allow: GET, POST, PUT, DELETE, OPTIONS
Date: Fri, 13 Mar 2026 13:00:00 GMT
Server: nginx
```
`Allow`가 핵심 헤더

## 상태코드
클라이언트가 보낸 요청의 처리 상태를 응답에서 알려주는 기능
클라이언트 입장에서 상태코드를 모르면 200대, 400대, 500대 큰 범위로 나누어 처리하면 된다.

* 100 Continue
  * 큰 요청 body를 보내기 전에 서버가 요청을 받아도 되는지 확인하는 것이다.

100 Continue 응답이 필요한 상황
```text
POST /upload HTTP/1.1
Content-Length: 500[json타입.md](../database/mysql/json%ED%83%80%EC%9E%85.md)MB
```
큰 파일을 업로드해야 하는 상황에서 서버가 요청을 거절한다면 (인증 실패, 권한 없음, 요청 형식 오류 등)
클라이언트는 불필요하게 500MB를 전송하는 **네트워크 낭비**가 발생한다.

```text
POST /upload HTTP/1.1
Host: example.com
Content-Length: 524288000
Expect: 100-continue
```
```text
HTTP/1.1 100 Continue
```
클라이언트는 body없이 헤더만 먼저 보내고 (100-continue 값이 담긴 Expect 헤더도 함께)
서버로부터 100 Continue 응답을 받으면 그때 파일을 업로드한다.
만약 서버가 거절한다면 body를 보내지 않는다. (서버는 100 Continue 응답 또는 에러 코드로만 답해야 한다)


* 200~299: 성공 상태 코드
  * 200 OK: 요청은 정상이며 본문에 요청된 리소스를 포함한다.
  * 201 CREATED: 응답은 생성된 리소스에 대한 참조가 담긴 Location 헤더와 함께 그 리소스를 참조할 수 있는 URL을 본문에 포함해야 한다.
  * 202 Accepted: 요청은 받아들여졌으나 처리가 완료되지 않았음 (요청 접수 후 1시간 뒤에 배치 프로세스가 요청을 처리하는 경우 등에서 사용)


* 300~399: 리다이렉션 상태 코드

리소스가 옮겨졌을 때, 클라이언트에게 리소스가 옮겨졌으며 어디서 찾을 수 있는지 알려주기 위해 
리다이렉션 상태 코드와 Location 헤더를 보낼 수 있다.
웹 브라우저는 3xx 응답 결과에 Location 헤더가 있으면 Location 위치로 자동 이동한다. (리다이렉트)

### 영구 리다이렉션 301, 308
* 리소스의 URI가 영구적으로 이동
* 원래의 URL을 사용하지 않는다. 검색 엔진 등에서도 변경을 인지한다.

### 301 Moved Permanently

리다이렉트 시 요청 메서드가 GET으로 변하고 본문이 제거될 수 있음 (maybe)
```text
POST /event HTTP/1.1
Host: localhost:8080

name=hello&age=20
```
```text
HTTP/1.1 301 Moved Permanently
Location: /new-event
```

```text
GET /new-event HTTP/1.1
Host: localhost:8080
```
자동 리다이렉트 될 때 GET으로 변경되어 요청될 수 있다. (일반적으로 URL이 바뀌면 내부적으로 전달해야하는 데이터가 다 바뀌기 때문에 웬만하면 GET으로 요청하는게 좋다.)
308에서는 POST가 유지된다.

### 일시적인 리다이렉션 302, 307, 303
* 리소스의 URI가 일시적으로 변경
* 따라서 검색 엔진 등에서 URL을 변경하면 안됨

| 상태코드                       | 메서드 변경                  | 의미                  |
| -------------------------- | ----------------------- | ------------------- |
| **302 Found**              | 브라우저가 **GET으로 바꿀 수 있음** | 옛날 방식 리다이렉트         |
| **303 See Other**          | **무조건 GET으로 변경**        | POST 처리 후 결과 페이지 이동 |
| **307 Temporary Redirect** | **메서드 유지**              | POST → POST 유지      |

* 302 Found
  * 원래 HTTP 스펙에서는 메서드를 유지해야 하지만 (POST → POST), 브라우저들이 멋대로 POST → GET으로 동작하는 경우가 있어 혼란이 생겼다.
* 303 See Other
  * POST → GET 로 명확히 바꾸는 상태 코드가 추가되었다.
* 307 Temporary Redirect
  * 명확하게 요청 메서드와 본문을 유지하는 상태 코드가 추가되었다. (POST → POST, GET → GET)

* 400~499: 클라이언트 에러 상태 코드

| 상태코드 | 이름           | 의미            | 설명 / 예시                                       |
| ---- | ------------ | ------------- |-----------------------------------------------|
| 400  | Bad Request  | 요청 구문, 메시지 오류 | 클라이언트는 요청 내용을 다시 검토하고 보내야 함 (요청 파라미터가 잘못된 경우) |
| 401  | Unauthorized | 인증 필요         | `WWW-Authenticate` 헤더로 인증 방법 전달               |
| 403  | Forbidden    | 권한 부족         | 인증은 됐지만 접근 권한 없음                              |
| 404  | Not Found    | 리소스 없음        | 리소스가 없거나 숨기기 위해 사용                            |

* 500~599: 서버 에러 상태 코드

| 상태코드 | 이름                    | 의미                                      |
|------|-----------------------|-----------------------------------------|
| 500  | Internal Server Error | 서버가 요청을 처리할 수 없게 만드는 에러를 만났을 때 사용       |
| 501  | Not Implemented       | 서버가 지원하지 않는 메서드를 사용한 경우                 |
| 503  | Service Unavailable   | 서비스 이용불가, 서버가 일시적인 과부하로 잠시 요청을 처리할 수 없음 |
| 504  | Gateway Timeout       | 리소스 없음                                  |

504 상태코드
```text
Client → Gateway(또는 Load Balancer) → Backend Server
```
일반적인 구조는 위와 같다.
여기서 Gateway → Backend 로 요청을 보냈는데 백엔드가 너무 오래걸려서 응답이 오지 않아 Gateway가 대신 응답한다.

# 4장 커넥션 관리
## TCP 커넥션
전 세계 모든 HTTP 통신은 TCP/IP를 통해 이루어진다.
따라서 연결이 수립된 이후에 데이터를 전달한다.
TCP/IP 커넥션이 맺어지면 클라이언트와 서버 간 주고받는 메시지들은 손실되거나 손상되지 않고 순서가 바뀌지 않고 전달됨이 보장된다.

1. 프로그램이 Hello World 메시지 생성
2. SOCKET 라이브러리를 통해 전달
3. TCP 정보 생성
4. IP 패킷 생성, TCP 데이터 포함

TCP는 데이터 스트림을 메시지 데이터를 포함한 TCP 세그먼트 단위로 잘게 나누고
IP 패킷에 담아 데이터를 전달한다.

TCP/IP 패킷
* IP 패킷
  * 출발지 IP, 목적지 IP
* TCP 세그먼트
  * 출발지 PORT, 목적지 PORT, 전송 제어, 순서, 검증 정보
  * 전송 데이터

### TCP 소켓 프로그래밍
운영체제는 TCP 커넥션 관련 여러 기능을 제공한다.
소켓 API를 사용하면 TCP endpoint를 생성하고 데이터 스트림을 읽고 쓸 수 있다.

```text
s1: 새로운 소켓을 만든다.
s2: 80 포트로 소켓을 묶는다.
s3: 소켓 커넥션을 허가한다
s4: 커넥션을 기다린다.

c1: IP 주소와 포트를 얻는다.
c2: 새로운 소켓을 생성한다.
c3: 서버의 IP 포트로 연결한다.

s5: 애플리케이션 커넥션 통지
s6: 요청을 일기 시작한다.

c4: 성공적으로 연결
c5: HTTP 요청을 보낸다.
c6: HTTP 응답을 기다린다.

s7: HTTP 요청 메시지를 처리한다.
s8: HTTP 응답을 보낸다.
s9: 커넥션을 닫는다.

c7: HTTP 응답을 처리한다.
c8: 커넥션을 닫는다.
```

### TCP 관련 지연
(1) TCP 3 way handshake
1. SYN
2. SYN + ACK
3. ACK
RTT(Round Trip Time) 만큼의 네트워크 지연이 발생한다.

(2) 확인 응답 지연
세그먼트를 받으면 확인 응답 패킷을 송신자에게 반환한다.
(송신자가 특정 시간 안에 확인 응답 메시지를 받지 못하면 패킷이 파기되었거나 오류가 있는것으로 판단하고 데이터를 다시 전송한다.)
TCP는 네트워크 효율을 위해 ACK를 즉시 보내지 않고 잠시 기다렸다가 한번에 보내는 경우가 있다.
이 과정에서 수십 ms 정도의 지연이 발생할 수 있다.

(3) 느린 시작 (TCP Slow Start)
TCP 데이터 전송 속도는 TCP 커넥션이 만들어진지 얼마나 지났는지에 따라 달라진다.
데이터가 성공적으로 전송됨에 따라 속도를 높여나간다. 
(처음부터 데이터를 많이 보내면 네트워크 혼잡이 발생할 수 있으니 천천히 보내면서 네트워크 상태 확인, 네트워크 혼잡으로 인한 패킷 손실(유실)을 막기 위해 사용한다)

(4) 네이글 알고리즘
작은 패킷들이 네트워크를 장악하여 성능을 떨어뜨리는 것을 방지하기 위해 설계된 알고리즘
전송할 데이터가 최대 세그먼트 크기만큼 쌓일 때까지 기다린다. 
만약 크기가 작더라도 이전에 보낸 패킷에 대한 확인 응답을 받았다면 즉시 보낸다.

HTTP와 같이 작은 메시지를 주고받는 상호작용형 서비스에서는 확인 응답이 올 때까지 데이터를 보내지 않고 대기하므로 심각한 지연이 발생할 수 있다.
최신 웹 서버에서는 이 알고리즘을 끄기 위해 TCP_NODELAY 옵션을 설정한다.

(5) TIME_WAIT의 누적과 포트 고갈
TCP 연결을 끊을 때 마지막 ACK를 보내는 쪽(클라이언트)은 다음 연결의 데이터와 섞이는 것을 방지하고 ACK가 유실되었을 때 재전송하기 위해
일정 시간(약 2분)동안 커넥션을 닫지 않고 유지한다.
커넥션은 종료되었지만 포트가 TIME_WAIT 상태에 묶여 재사용할 수 없게 된다.

### Connection 헤더
Connection 헤더는 현재 연결에서만 유효한 옵션을 알려주는 용도로 사용된다.
다음 서버에는 전달하지 말라고 표시하는 차단 리스트라고 할 수 있다.

```text
Client → Proxy → Server

Client ↔ Proxy  (연결 A)
Proxy ↔ Server  (연결 B)
```
A랑 B는 완전히 다른 TCP 연결이다.
Proxy가 새로운 요청을 생성하여 다시 서버에 전달한다.
기본 헤더는 복사하여 그대로 전달하지만 Connection 관련 헤더는 제거한다.

ex)
```text
Connection: keep-alive, X-Secret
X-Secret: hello
```
`Connection`에 명시된 헤더는 커스텀이든 뭐든 전부 제거된다. (Connection, keep-alive, X-Secret)
해당 연결에서만 사용되고 다음으로 넘어가지 않는다.

### 순차적인 트랜잭션 처리에 의한 지연
이미지가 3개 포함된 웹 페이지가 있다고 가정하자.
```text
1. HTML 요청 → 응답
2. 이미지1 요청 → 응답
3. 이미지2 요청 → 응답
4. 이미지3 요청 → 응답
```
이처럼 리소스를 순차적으로 요청하면 총 4개의 HTTP 트랜잭션이 발생한다.
이 경우 각 요청마다 새로운 TCP 연결이 생성되고 느린 시작(TCP Slow Start)가 반복되어 수행되기 때문에 전체 응답 속도가 크게 저하된다.
※ 느린 시작(TCP Slow Start): 네트워크 혼잡을 방지하기 위해 처음에는 적은 양의 데이터를 전송하고 점차 전송량을 증가시키는 방식
이러한 문제를 해결하고 HTTP 커넥션의 성능을 향상시키기 위해 다음과 같은 방법들이 사용된다.

**(1) 병렬 커넥션**
```text
1. HTML 요청 → 응답

(이후 동시에)
2. 이미지1 요청
2. 이미지2 요청
2. 이미지3 요청
```
병렬 커넥션은 여러 개의 TCP 연결을 동시에 생성하여 여러 HTTP 요청을 병렬로 처리하는 방식이다.
이를 통해 각 요청이 독립적인 TCP 연결에서 처리되며 TCP Slow Start가 동시에 진행되어 전체 응답 시간이 단축된다.

병렬 커넥션은 항상 성능을 향상시키는 것은 아니다.
네트워크 대역폭이 제한적인 환경에서는 여러 요청을 동시에 보내더라도 오히려 네트워크 혼잡이 발생해 성능이 저하될 수 있다.

또한 클라이언트가 너무 많은 TCP 커넥션(예: 100개)을 동시에 생성하면 서버에 큰 부하를 줄 수 있다.
이러한 이유로 대부분의 브라우저는 도메인당 생성할 수 있는 병렬 커넥션 수를 제한한다.

**(2) 지속 커넥션**
웹 클라이언트는 보통 같은 사이트에 여러 개의 커넥션을 맺고 요청을 반복적으로 보낸다. (사이트 지역성이라고 한다)
따라서 HTTP/1.1을 지원하는 기기는 요청이 처리가 완료된 후에도 TCP 커넥션을 유지하여 앞으로 있을 HTTP 요청에 재사용할 수 있다.
이 방식은 매 요청마다 TCP 연결을 새로 생성하지 않아도 되므로 3-way handshake에 소요되는 시간을 절약할 수 있다.
또한 이미 맺어져 있는 커넥션을 재사용하기 때문에 TCP 느린 시작(TCP Slow Start)로 인한 초기 전송 지연을 피함으로써 더 빠르게 데이터를 전송할 수 있다.

하지만 지속 커넥션을 잘못 관리할 경우 계속 연결된 상태로 있는 수많은 커넥션이 쌓이게 될 수 있다.

=> 병렬 커넥션과 지속 커넥션을 동시에 사용한다. (웹 애플리케이션은 적은 수의 병렬 커넥션을 맺고 유지한다)

### HTTP/1.0의 Keep-Alive 커넥션
HTTP/1.0에서는 기본적으로 요청-응답이 끝나면 TCP 커넥션을 종료한다.
하지만 Keep-Alive를 사용하면 하나의 TCP 커넥션을 여러 HTTP 요청에 재사용할 수 있어, 커넥션을 맺고 끊는 데 드는 비용을 줄일 수 있다.

Keep-Alive는 클라이언트와 서버 사이의 TCP 연결을 재사용하는 기술이다. (같은 TCP endpoint (IP + Port))
(TCP handshake 감소, 서버 connection 생성 비용 감소, latency 감소)
단 connection을 오래 잡고 있기 때문에 timeout, max request 같은 제한을 둬야한다.

클라이언트는 요청 시 Connection: Keep-Alive 헤더를 포함하여 지속 커넥션 사용 의사를 전달한다.
이 요청을 받은 서버가 이를 지원할 경우, 응답에도 동일하게 Connection: Keep-Alive 헤더를 포함하여 커넥션을 유지하겠다는 의사를 표시한다.
클라이언트는 응답에 Connection: Keep-Alive 헤더가 포함되어 있지 않다면, 서버가 Keep-Alive를 지원하지 않는 것으로 판단하고 커넥션이 종료될 것을 예상한다.

응답 헤더 예시
```text
Connection: Keep-Alive
Keep-Alive: max=5, timeout=120
```
timeout: 커넥션을 유지하는 최대 시간 (보장은 아님)
max: 해당 커넥션으로 처리할 수 있는 최대 HTTP 트랜잭션 수 (보장은 아님)

특징 및 주의사항
HTTP/1.0에서는 Keep-Alive가 기본 동작이 아니기 때문에, 지속 커넥션을 사용하려면 반드시 Connection: Keep-Alive 헤더를 명시해야 한다.
또한 커넥션을 계속 유지하려면 모든 요청 메시지에 Connection: Keep-Alive 헤더를 포함해야 하며
해당 헤더가 없을 경우 서버는 요청 처리 후 커넥션을 종료할 수 있다.

커넥션을 계속 유지하려면, 각 HTTP 응답이 어디서 끝나는지 알 수 있어야 한다.
그래야 다음 응답과 구분할 수 있기 때문이다.
이를 위해 `Content-Length`로 길이를 알려주거나 청크 전송 방식처럼 데이터의 끝을 명확하게 표시해야 한다.
만약 응답의 끝을 알 수 없다면 연결을 끊어서 구분할 수밖에 없기 때문에 Keep-Alive를 사용할 수 없다.

### Keep-Alive와 멍청한 프락시
프락시가 Connection 헤더 의미를 몰라서 Keep-Alive를 깨버리는 문제
Connection 헤더는 hop-by-hop으로 Client ↔ Proxy 사이에서만 유효하다.
프록시가 각 연결마다 새로 결정해서 설정해야 한다.

```text
Client → Proxy → Server
```

[1] 정상적인 Keep-Alive 흐름

클라이언트 요청
```text
Connection: Keep-Alive
```
프록시는 클라이언트로부터 `Connection: Keep-Alive` 헤더를 받으면 이를 그대로 전달하지 않고 제거한다.
이후 서버로 요청을 보낼 때는 프록시가 판단하여 커넥션을 유지할지 여부를 결정하고 필요에 따라 `Connection` 헤더를 새로 설정한다.
응답을 클라이언트에게 전달할 때도 마찬가지로 클라이언트와의 연결을 유지할지 여부를 프록시가 판단하여 `Connection` 헤더를 다시 설정한다.

[2] 멍청한 프록시
Connection 헤더가 hop-by-hop이라는 사실을 이해하지 못하는 프록시를 ‘멍청한 프록시’라고 한다.
클라이언트로부터 `Connection: Keep-Alive` 헤더를 받으면 이를 제거하지 않고 서버에 그대로 전달 후
서버로 부터 받은 응답 헤더 `Connection: Keep-Alive` 역시 그대로 클라이언트에게 전달한다.
클라이언트는 해당 연결이 유지된다고 생각하고 다음 요청을 보내지만 실제로는 연결이 이미 종료되었거나 재사용할 수 없는 상태일 수 있어 오류가 발생할 수 있다.

[2] 멍청한 프록시를 해결하기 위한 `Proxy-Connection` 비표준 헤더
클라이언트는 `Proxy-Connection: Keep-Alive` 헤더를 보낸다.
멍청한 프록시는 서버에  `Proxy-Connection: Keep-Alive` 헤더를 그대로 전달하기 때문에 서버는 이 헤더를 무시하면서
`Connection` 헤더를 클라이언트에게 내려주지 않게된다.

영리한 프록시는 `Proxy-Connection` 헤더를 인식하고 클라이언트와 keep-alive로 통신하는데 동의하며
서버와 keep-alive 커넥션을 맺을 수도 있고 맺지 않을 수도 있다.
다만 Proxy-Connection은 프록시가 하나일 때만 그나마 예측 가능하고 여러 개면 믿고 쓰면 안 되는 헤더다.
(멍청한 프록시가 포함되어있다면 다시 문제가 발생할 수 있다.)

### HTTP/1.1의 지속 커넥션
HTTP/1.0의 keep-alive 커넥션과 달리 HTTP/1.1의 지속 커넥션은 기본적으로 활성화되어있다.
HTTP/1.1에서는 별도 설정을 하지 않는 한 모든 커넥션을 지속 커넥션으로 취급한다.
HTTP/1.1 애플리케이션(서버/클라이언트)은 트랜잭션이 끝난 다음 커넥션을 끊으려면 `Connection: close` 헤더를 명시해야 한다.

지속 커넥션을 사용하려면 모든 메시지가 자신의 길이 정보를 정확히 가지고 있어야 한다.
이를 위해 `Content-Length`로 길이를 알려주거나 청크 전송 방식처럼 데이터의 끝을 명확하게 표시해야 한다.

### 파이프라인 커넥션
HTTP/1.1은 지속 커넥션을 통해서 요청을 파이프라이닝할 수 있다.
연결을 계속 유지할 수 있어서 그 위에서 요청을 끊지 않고 연속으로 보낼 수 있는것이다.
파이프라이닝: 연결 위에서 응답을 기다리지 않고 요청을 연속으로 보내는 것

파이프라이닝 제약사항
* HTTP 클라이언트는 커넥션이 지속 커넥션인지 확인하기 전까지는 파이프라인을 이어서는 안된다.
  * 연결이 계속 유지되는지 모르는데 요청을 연속으로 보내면 안된다.
  * 서버가 응답1을 보내고 연결 끊으면 요청2, 요청3이 날아가게 된다.
* HTTP 응답은 요청 순서와 같게 와야한다. HTTP 메시지는 순번이 매겨져있지 않아 응답이 순서 없이 오면 순서에 맞게 정렬시킬 수 없다.
  * 요청1 → 요청2 → 요청3, 응답1 → 응답2 → 응답3
  * TCP에서 순서를 보장해준다. (Sequence Number를 사용해 순서에 맞게 전달)
* HTTP 클라이언트는 커넥션이 언제 끊어지더라도 완료되지 않은 요청이 파이프라인에 있으면 언제든 다시 보낼 준비가 되어있어야한다.
  * 서버가 응답1 보내고 연결 끊어버리면 클라이언트는 안 끝난 요청 요청2, 요청3은 다시 보내야한다.
* POST 요청같이 반복해서 보낼 경우 문제가 생기는 요청은 파이프라인을 통해 보내면 안된다.
  * 재전송해도 문제 없는 멱등한 요청만 파이프라인에 넣어야 한다 (GET, PUT, DELETE)

### 우아한 커넥션 끊기
커넥션은 에러가 없더라도 언제든 끊을 수 있다.
데이터를 다 주고받은 뒤 안전하게 TCP 연결을 종료하는 것
Server가 응답 보내는 중에 Client가 갑자기 연결 끊는 경우 데이터가 일부 유실될 수 있다.

```text
Client → FIN → Server
Server → ACK → Client
Server → FIN → Client
Client → ACK → Server
```
```text
1. 요청/응답 완료
2. 한쪽이 FIN (나는 더 이상 안 보냄)
3. 상대방이 남은 데이터 다 보냄
4. 마지막 ACK 후 종료
```
우아한 커넥션 종료는 데이터 손실 없이 안전하게 연결을 끊는 것을 의미하며 TCP에서는 이를 위해 4-way handshake를 사용한다.

```text
1. Server → ACK   (일단 FIN 받았다고 응답)
2. ...남은 데이터 전송...
3. Server → FIN   (이제 나도 끝내겠다고 알림)
```
서버가 아직 보낼 데이터가 남아있을 수 있기 때문에 3-way로 끝낼 수 없다.

커넥션의 출력 채널을 먼저 끊는 것이 안전하다. (데이터 전송은 못하지만 ACK/FIN은 가능)
응답 데이터가 잘 도착했어도 이미 커넥션이 끊겨있는 경우 `connection reset by peer` 에러를 낸다.

HTTP/1.1 → KeepAlive
HTTP/2 → multiplexing (동시 요청)

# 5장 웹서버
모든 웹 서버는 리소스에 대한 HTTP 요청을 받아서 콘텐츠를 클라이언트에게 돌려준다.

웹서버가 하는 일 
* 요청 수신
  * 클라이언트로부터 HTTP 요청을 받는다
  * TCP 연결을 통해 데이터 수신
* 요청 메시지 파싱
* 리소스 식별
  * 정적 파일 (HTML, 이미지 등)
  * 동적 콘텐츠 (API, 서버 로직)
* 응답 생성
* 응답 전송
* 커넥션 관리
  * 연결을 어떻게 처리할지 결정
    * Keep-Alive 유지
    * Connection: close로 종료

# 6장 프락시
프락시는 클라이언트 ↔ 서버 사이에서 요청/응답을 대신 전달하는 중개자
클라이언트 대신 서버에 요청하고 서버 응답을 다시 클라이언트에게 전달한다.

**프락시 vs 게이트웨이**
프락시는 같은 프로토콜끼리 연결한다. (HTTP ↔ HTTP, 중개자(같은 언어))
게이트웨이는 다른 프로토콜끼리 연결한다. (HTTP ↔ FTP, HTTP ↔ SMTP, 번역기(다른 언어))

실제로는 완전히 구분이 어렵고 경계가 모호하다.
프락시도 HTTP 버전 차이 때문에 약간의 변환을 수행한다.

**왜 프락시를 사용할까?**
* 보안
* 성능
* 비용 절감

프락시는 모든 HTTP 트래픽을 관찰, 제어 가능하다.

* 보안 & 접근 제어
  * 어린이 필터 → 유해 사이트 차단
  * 접근 제어 → 특정 사용자/회사만 허용
  * 방화벽 역할
* 캐싱
  * 자주 요청되는 데이터 저장
  * 서버 부하 감소 + 응답 속도 증가
* 리버스 프락시 (대리 프락시)
  * 서버 앞단에 위치하여 실제 서버를 숨긴다. 클라이언트는 프락시를 서버로 착각한다.
  * 로드 밸런싱, 보안(서버 직접 노출 방지)
* 콘텐츠 라우팅
  * 요청을 적절한 서버로 분배
  * 가까운 서버로 보내기(CDN 느낌)
* 트랜스 코딩(콘텐츠 변환)
  * 이미지 포맷 변환(GIF → JPG)
  * 데이터 압축
* 익명화 프락시 (개인정보 보호)
  * IP 주소 제거
  * 쿠키/Referer 제거
  
### 프락시 서버 배치
* 출구 프락시
  * 클라이언트 쪽(내부 네트워크)에 위치
  * 회사, 학교 내부 네트워크의 출구
  * 내부 사용자를 대신해서 밖으로 나가는 대리인
  * 역할: 내부 사용자 → 외부 인터넷 요청을 대신 처리
    * 트래픽 제어 (누가 어디 가는지 통제)
    * 콘텐츠 필터링 (유해 사이트 차단)
    * 캐싱 (같은 요청 빠르게 처리)
* 입구 프락시
  * 인터넷 초입(네트워크 경계)에 위치
  * ISP(통신사) 쪽 또는 네트워크 진입 지점
  * 인터넷 들어오는 입구에서 트래픽 최적화 담당
  * 역할: 외부 사용자 요청을 받아서 효율적으로 처리
    * 캐싱 → 다운로드 속도 향상
    * 캐시 프락시 사용
* 대리 프락시
  * 서버 앞단에 위치 (웹 서버 바로 앞)
  * 서버를 대신해서 응답하는 가짜 서버
  * 역할: 클라이언트 요청을 받아서 뒤에 있는 서버로 전달
  * 클라이언트는 프락시를 서버라고 착각하고 실제 서버는 숨겨진다.
    * 로드밸런싱
    * 보안 (서버 직접 노출 방지)
    * SSL 처리
    * 캐싱

### 프락시 계층
```text
클라이언트 → 프락시1 → 프락시2 → 프락시3 → 서버
```
프락시는 여러 개가 연결된 체인(계층)으로 동작이 가능하다.
자식 프락시: 클라이언트에 가까운 프락시 (아웃바운드)
부모 프락시: 서버에 가까운 프락시 (인바운드)

**프락시 계층이 하는 일**
* 캐시 활용 (CDN 느낌)
  * 가까운 캐시 서버에서 응답, 없으면 원 서버 요청하여 속도 개선 및 트래픽 절감
* 특화 처리 (요청 종류별로 역할 분리)
  * 이미지 요청 → 이미지 압축 프락시, 동영상 → 스트리밍 최적화 프락시
* 부하 분산
  * 자식 프락시가 여러 부모 중 선택 (현재 부하 상태 및 서버 위치 기준)
* 라우팅
  * URI 기반으로 다른 프락시/서버 선택 (`/image` → 이미지 서버, `/api` → API 서버)

### 프락시는 어떻게 트래픽을 가로챌까?
* 클라이언트 설정
  * 브라우저에서 프락시 직접 설정 (가장 기본적인 방법)
* 네트워크 레벨
  * 라우터/스위치가 강제로 프락시로 전달
* DNS 변경
  * 도메인을 프락시로 가리키게 설정(클라이언트는 모르게 프락시 사용)
* 서버 리다이렉트
  * 서버가 프락시로 다시 가라고 응답(HTTP 3xx 사용)

클라이언트 설정 방식은 다시 수동 설정, 자동 설정으로 나뉜다.
* 수동 설정: 프락시 IP + 포트 직접 입력
단순하지만 유연성이 떨어진다 (하나만 지정 가능하다)

* 자동 설정
PAC (Proxy Auto Configuration) 이라는 JS 파일로 프락시 사용 여부 결정한다.
```JavaScript
function FindProxyForURL(url, host) {
    if (host == "internal.com") return "DIRECT";
    return "PROXY proxy.com:8080";
}
```
반환값 의미
DIRECT → 프락시 없이 직접 연결
PROXY host:port → 해당 프락시 사용
SOCKS host:port → SOCKS 프락시 사용

WPAD (자동 발견): 브라우저가 자동으로 PAC 파일 찾아 별도 설정 없이도 프락시 사용이 가능하다.
